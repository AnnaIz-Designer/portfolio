02

Token Architecture

Before the pipeline, I had to design the token system. Three tiers: primitive values → semantic roles → component tokens. The key insight: when a brand colour changes, only one primitive updates — everything else follows automatically.

Tier 1 · Primitives
Raw values. Never used directly in components.
Source only
Tier 2 · Semantic
Role-based aliases. Names describe meaning, not appearance.
★ Key layer
Tier 3 · Component
Scoped overrides. References semantic, never primitives.
Optional
primitives.css
/* Raw values */
--blue-900:  #181755;
--gray-100:  #ebebdf;
--space-16:  16px;
semantic.css — rebrand-proof
/* Meaning, not appearance */
--color-action:   var(--blue-900);
--color-surface:  var(--gray-100);
--space-component:var(--space-16);
component.css
/* References semantic only */
--btn-bg:      var(--color-action);
--card-padding:var(--space-component);

Decisions & Consequences

Each decision came with a tradeoff. Here's what I chose, and what actually happened as a result.

Decision 01
Semantic naming over descriptive naming
Rejected --blue-button. Chose --color-action. When the brand colour changes, descriptive names become false. Semantic names never need to change.
Naming · Scalability
What happened
We tested a full rebrand 6 months later. Zero token names needed updating — only the primitive values changed. The engineers noticed. It built trust in the whole system.
"The first version used names like 'sage-button'. The moment we tested a rebrand, every token name became a lie." — Engineer, iteration 2
Decision 02
PR review step — not direct push to main
I originally proposed auto-pushing to main on publish. Engineering vetoed it. I agreed: a 2-minute review is worth it for the safety net.
Safety · Engineering trust
What happened
Two wrong tokens were caught in the first month — a designer had accidentally edited the wrong spacing variable. Without the PR step, those changes would have shipped silently to production.
"That PR diff saved us twice. I can see exactly what changed before it goes anywhere." — Lead engineer
Decision 03
One Figma collection with modes — not one per platform
Separate collections for iOS, Android, Web = 5× maintenance. One collection + modes + Style Dictionary output transformation = one file to maintain.
Architecture · One source
What happened
Designers update one place. The pipeline handles five output formats. Three months in, no one has asked "which file do I edit?" — that question doesn't exist anymore.
Decision 04
Generated file is never hand-edited
tokens.css is 100% generated. README: "Do not edit this file." Engineers wanted inline overrides. I held the line: overrides live in overrides.css only.
Process · Source of truth
What happened
One engineer tried to add a manual override directly into tokens.css. It got overwritten on the next sync. After that, everyone used the override layer correctly. Sometimes the system needs to enforce itself.

What I'd do differently
I'd add a visual colour swatch diff in the PR body — old value and new value as swatches, not just hex strings. Typography tokens also stayed manual due to API limitations; that's the next thing to solve.
06

What I learned

The technical part of this project was straightforward once the architecture was settled. The hard part was the three weeks before any code was written — convincing engineering that a designer-owned automation touching their codebase was a good idea.

The resistance was reasonable. Designers don't usually write to repos. The concern wasn't capability — it was trust. What happens when something breaks? What happens when a designer accidentally changes the wrong variable and it goes straight to production?

The PR review step wasn't a technical requirement. It was a trust mechanism. By keeping engineers in the loop for 2 minutes, they went from skeptical to fully bought in. That buy-in is what made the whole thing work.

The API limitation with typography tokens was the one thing I couldn't solve. Font family, size, line height — none of it was exportable via the Figma Variables REST API at the time. I documented the boundary clearly and moved on. A partial solution that works completely is better than a complete solution that doesn't ship.

The naming convention alignment across 3 departments took 4 weeks. Not because anyone was being difficult — but because iOS, Android, and Web engineers all had legitimate conventions that conflicted with each other. In the end we agreed on a source format and let Style Dictionary translate. The right answer was to solve it at the transformation layer, not the naming layer.
